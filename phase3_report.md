# AutoDevOps: 管理ダッシュボードとFirestoreリアルタイム連携 (Phase 3)

このドキュメントでは、Phase 3 で構築した「AIが検知したエラー（インシデント）を管理者がブラウザ上で確認でき、Firestoreとリアルタイムに同期するフロントエンド管理ダッシュボード（Flutter）」の仕組みと実行方法を解説します。

---

## 🌟 概要：何ができるのか？

Flutterで作成したWebダッシュボード（`frontend_flutter`）を起動すると、以下の処理と連携が**すべて自動・リアルタイム**で行われます。

1. **リアルタイムデータ同期 (Firestore 連携)**:
   - バックエンド（`pipeline.py`等）が検知してFirestoreのデータベースに書き込んだインシデント（エラー）情報を、アプリが自動で監視します。
   - データベースに新しいエラーが追加されたりステータスが更新されたりすると、**画面をリロードしなくても数秒で自動的に画面が更新**されます。
2. **デモモードと実データモードの切り替え**:
   - 画面上の「DEMO MODE」スイッチを切り替えるだけで、ローカルのダミーデータ表示（デモモード）と、実際のデータベース（Firestore）から取得した実データ表示（実データモード）を瞬時に切り替えられます。
3. **堅牢なエラーハンドリング**:
   - ネットワークの切断や権限エラーでFirestoreとの通信が切れた場合でも、アプリがクラッシュ（フリーズ）しない設計になっています。
   - エラーが発生した際は、画面下部に赤いメッセージ（SnackBar）でエラー内容を分かりやすくユーザーに通知します。

---

## 🛠️ 事前準備（初回のみ）

フロントエンド（Flutter Web）をローカルで動かすには、以下の設定が必要です。

### 1. Flutter の環境を準備する
お使いのPCに Flutter SDK がインストールされていることを確認してください。

### 2. 必要なファイルを配置する
*   `frontend_flutter/lib/firebase_options.dart` : FlutterアプリがFirebase（Firestore）に接続するための設定ファイルです（事前に用意したものをこの位置にコピーして配置します）。

### 3. パッケージをインストールする
ターミナルで `frontend_flutter` フォルダに移動し、必要なパッケージをダウンロードします。
```bash
cd frontend_flutter
flutter pub get
```

### 4. Firestore セキュリティルールのデプロイ（管理者用）
アプリからFirestoreにアクセスできるよう、セキュリティルールをFirebaseプロジェクトに反映します。
```bash
# プロジェクトのルートディレクトリで実行
firebase deploy --only firestore:rules
```

---

## 🚀 実行方法

準備ができたら、ターミナルで以下のコマンドを実行してWebブラウザ（Chromeなど）でアプリを起動します。

```bash
# frontend_flutter フォルダに移動（すでに移動している場合は不要）
cd frontend_flutter

# Webアプリとして起動（Chromeで開く）
flutter run -d chrome
```

起動が完了すると、ブラウザが自動的に立ち上がり、ダッシュボード画面が表示されます。

---

## ✅ 確認方法（動作確認の手順）

1. **デモモードの切り替え**
   - 画面右上（または設定エリア）にある「DEMO MODE」のスイッチを **OFF** にします。
2. **実データのリアルタイム表示**
   - スイッチをOFFにすると、Firestoreから取得した本物のインシデントデータ（例: `user-service` で検知された未解決のエラーなど）が一覧に表示されます。
3. **エラー表示の確認（接続失敗時）**
   - もしFirebaseに正しく接続できない、あるいは権限が足りない場合、画面下部に赤い通知バー（SnackBar）が表示され、「Firestoreの取得に失敗しました」等のエラーメッセージが表示されます。

---

## 💡 よくあるエラーと対処法

*   **エラー**: 画面下部に `[FirebaseException] ... permission-denied` と表示される
    *   **原因**: Firestoreのセキュリティルールでアクセスが許可されていないか、ログイン認証が通っていません。
    *   **対処法**: `firebase deploy --only firestore:rules` が実行され、データベースのルールが正しくデプロイされているかを確認してください。
*   **エラー**: `firebase_options.dart` のインポートでエラーが出る、ビルドに失敗する
    *   **原因**: Firebaseの構成ファイルが `lib/` フォルダに存在しないか、内容が古いです。
    *   **対処法**: 正しい `firebase_options.dart` が `frontend_flutter/lib/` 配下にあるか確認してください。

## ⏭️ 次のステップ (Phase 4)

Phase 3 でフロントエンドとデータベースのリアルタイム連携が完了しました。
最終フェーズである **Phase 4（本番運用クオリティへのブラッシュアップ）** では、以下の機能を実装します。

1. **ユーザー認証（Firebase Authentication）の導入**:
   - 一般ユーザーは閲覧のみ、特定の管理者（エンジニア）だけが「承認（マージ）ボタン」を押せるようにセキュリティを制限します。
2. **サーキットブレーカーの導入（バックエンド）**:
   - 同じバグが連続して発生した際、AIが無限にループしてPRを作成し続けるのを防ぐストッパー機能をPythonバックエンド側に追加します。
3. **Diff画面と承認アクションの完全結合**:
   - UI上の「承認（マージ）」を押したときに、GitHubの該当Pull Requestが自動でマージされるAPI処理を結合させます。
