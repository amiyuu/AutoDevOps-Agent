### 開発ロードマップ（4つのフェーズ）

一気にすべてを実装するのではなく、段階を追ってスコープを拡大していくアプローチ（インクリメンタル開発）を推奨します。

#### Phase 1: ローカル検証＆AIパッチ精度の検証（最優先）

ゴール: ローカル環境で「エラーログ」と「ソースコード」を与えた際、LLMが正しい修正コードを安定して出力できるか検証する。

タスク:

[ ] サンプルとして「わざとエラーが出るコード（Syntax ErrorやNull Pointerなど）」を用意する。

[ ] OpenAI API / Vertex AI等を用いたバグ分析スクリプト（Python/Node.js）をローカルに作成。

[ ] LLMの出力を「プログラムがパースしやすいJSON形式（Structured Outputs等）」に固定するプロンプトの調整。

#### Phase 2: エラー検知とGitHub/Firebaseパイプライン構築

ゴール: 本番エラーの発生から、自動でGitHubのPR作成、Firebaseへのデータ保存までをノンストップで繋ぐ。

タスク:

[ ] Cloud LoggingからPub/Sub経由で起動するCloud Functions（Backend）のガワを作成。

[ ] GitHub Appの作成と、リポジトリへのアクセス権限（コード読み書き、PR作成）の設定。

[ ] BackendにGitHub API（Octokitなど）を組み込み、自動ブランチ作成・コミット・PR作成を実装。

[ ] Firestoreのデータベース設計を行い、Backendからデータを保存する処理を実装。

#### Phase 3: フロントエンドUIとDiff表示・承認機能の実装

ゴール: 管理画面で「どこがどう直るか」を赤緑で確認でき、ボタン一つでマージが実行できる。

タスク:

[ ] フロントエンド（React, Next.js, Vueなど）の新規プロジェクト立ち上げ。

[ ] Firebase SDKを導入し、Firestoreの変更をリアルタイムに検知（onSnapshot）して一覧表示。

[ ] 差分表示ライブラリ（例: react-diff-viewer）を組み込み、元コードと修正後コードのDiffをレンダリング。

[ ] Backendに「マージ実行API」を実装し、フロントの「承認ボタン」から呼び出せるようにする。

#### Phase 4: 本番運用クオリティへのブラッシュアップ

ゴール: 無限ループやセキュリティリスクを排除し、安心して本番運用できる状態にする。

タスク:

[ ] サーキットブレーカー（同一エラーに対する再試行回数の制限）の実装。

[ ] 大規模コード対応（エラー箇所周辺のファイルだけを抽出するロジックやAST解析の導入）。

[ ] ユーザー認証（Firebase Authentication）を導入し、承認ボタンを特定の管理者のみが押せるように制限。

#### 困難なポイントの具体的解決アプローチ（テクニカルガイド）

💡 対策1: LLMに「壊れないDiff/コード」を出力させる
LLMに git diff 形式（+ や - を含む形式）で出力させると、インデントのズレ等でプログラムがパースできなくなる確率が非常に高いです。

解決策: LLMには 「修正後のファイル全体のコード」 を丸ごと出力させます。フロントエンドのDiff表示やGitHubへのコミットは、Backend/フロントのライブラリ側でその文字列同士を比較して自動生成するのが最も安全です。

💡 対策2: コンテキスト制限（トークン節約＆精度向上）
リポジトリ全体の全ファイルをLLMに渡すと、トークン代が高騰し、関係のないコードがノイズになって精度が落ちます。

解決策:

スタックトレースからエラーが起きた「ファイルパス」を特定する。

そのファイルと、そのファイルが import している直近の依存ファイル（最大2〜3ファイル）のみを特定して、LLMへのインプットとする。

💡 対策3: 自動化の無限ループを防ぐ
修正コードをデプロイ ➔ 新たなバグ発生 ➔ 再度修正 ➔ さらにバグ発生……というループを防ぐ必要があります。

解決策: Firestore等で、過去1時間に作成された自動PRの数をカウントする仕組みを入れます。「1時間に3回以上自動修正が走った場合は、システムを強制停止（サーキットブレイク）し、Slack等で人間に強制的アラートを飛ばす」ロジックを必ず挟んでください。

💡 対策4: セキュリティとGitHub権限
万が一AIエージェントの認証情報が漏洩した場合、リポジトリ全体が乗っ取られる危険性があります。

解決策: 個人のGitHubアカウントに紐づくパーソナルアクセストークン（PAT）は絶対に使用せず、「GitHub App」を作成してください。特定の「対象リポジトリ」に対してのみ、かつ「Contents: Read & Write」「Pull Requests: Read & Write」の最小権限だけを許可して運用します。
